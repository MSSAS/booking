# Логика подготовки данных для сегментации клиентов санатория

## Исходные данные

**Файл:** `silver.csv` — 565 816 строк, 24 колонки.
Каждая строка — одна запись о госте в рамках одного бронирования.

**Ключевые понятия:**
- `grp_norm` — ID бронирования. Под одним бронированием могут быть несколько гостей (семья, корпоративная группа).
- `guest_id` — ID конкретного гостя. Один гость может иметь несколько бронирований за историю.
- `is_valid_for_rfm` — флаг наличия персональных данных гостя (1 = известный гость, 0 = аноним).

---

## Этап 1 — Фильтрация по is_valid_for_rfm

**Решение:** оставляем только строки с `is_valid_for_rfm == 1`.

**Почему:** задача — сегментация клиентов. Нам нужны гости, у которых есть `guest_id` и персональные данные (возраст, пол, история визитов). Анонимы (44 673 строки, 7.9%) не имеют этих атрибутов и не могут быть профилированы как клиенты. Они будут использованы в модели прогноза спроса, где важна сама загрузка, а не личность гостя.

**Результат:** 521 143 строки.

---

## Этап 2 — Понимание структуры дублей внутри grp_norm

Внутри одного бронирования (`grp_norm`) строк может быть несколько по двум принципиально разным причинам:

**Причина A — несколько гостей в одном бронировании (разные `guest_id`):**
Семья, пара или корпоративная группа едут вместе. Это норма. Пример: Фимин (1828) и Фимина (1829) под `grp_norm=294681`. Это два отдельных клиента — оставляем как две строки.

**Причина B — один гость с несколькими тарифными периодами (одинаковый `guest_id`):**
Технический артефакт системы бронирования. Один человек записан несколько раз внутри одного заезда из-за смены тарифа или продления пребывания. Пример: Фимин (1828) имеет 3 строки с датами 30.12→31.12, 31.12→03.01, 03.01→05.01. Это не три визита — это один заезд на 6 ночей, разбитый тарифными периодами.

**Как различаем:** группировка `groupby(['grp_norm', 'guest_id'])` — если count > 1, это причина B. Таких случаев 3 059.

## Этап 3 — Вычисление nights ДО схлопывания

**Решение:** `nights = normalize(check_out) - normalize(check_in)`, clip(lower=0).

**Почему до схлопывания:** один тарифный период — один промежуток дат. При схлопывании мы берём `sum(nights)` по всем периодам — так получаем реальную длительность заезда. Если бы мы вычисляли nights уже после схлопывания (от min check_in до max check_out), результат был бы тот же, но только при условии что периоды идут подряд без пропусков. Схема «до» — надёжнее.

**Почему normalize:** убираем время суток. Заезд в 14:00 и выезд в 12:00 следующего дня — это 1 ночь, а не 0.9 или 1.1.

**clip(lower=0):** на случай аномалий, где check_out < check_in (ошибки данных).

## Этап 4 — Распространение children внутри бронирования

**Проблема:** `adults`, `children` и `composition` — атрибуты номера, а не конкретного гостя. Они записаны только в одной строке из всех жильцов номера. Пример: у Фиминой стоит `adults=2`, у Фимина — `NaN`, хотя они в одном номере. Если не исправить, признак `has_children` будет занижен в 3 раза (39 449 гостей вместо ~132 027).

**Решение:** до схлопывания берём `transform('max')` по `grp_norm` — распространяем максимальное значение `children` на всех участников бронирования.

**Ограничение для крупных групп:** для бронирований > 10 гостей (корпоративные туры) не распространяем. В большой группе часть номеров может быть с детьми, часть — без. Приписывать всем 50 участникам `has_children=1` из-за двух детей в одном номере — неверно.

**Порог ≤ 10:** 88.5% бронирований с детьми — это 1–4 гостя (явные семьи). Порог с запасом покрывает расширенные семьи (бабушки, дедушки) и минимизирует ложные срабатывания.

**Результат:** has_children = 27% гостей (было 9%).

## Этап 5 — Схлопывание по (grp_norm, guest_id)

**Решение:** `groupby(['grp_norm', 'guest_id']).agg(...)`.

**Правила агрегации и обоснование:**

| Колонка | Агрегация | Обоснование |
|---|---|---|
| `check_in_dt` | `min` | Начало первого тарифного периода — реальный день заезда |
| `check_out_dt` | `max` | Конец последнего периода — реальный день выезда |
| `nights` | `sum` | Суммируем ночи по всем периодам → полная длительность заезда |
| `season` | `mode` | Берём преобладающий сезон (для многонедельных заездов) |
| `tariff` | `first non-null` | Берём первый непустой тариф — он описывает заезд в целом |
| `room_type` | `first non-null` | Аналогично — тип номера один для заезда |
| `children` | `max` | После этапа 4 — берём максимум (уже распространено) |
| `adults` | `max` | Аналогично |
| `guest_age` | `first` | Атрибут гостя из CRM, не меняется между заездами (проверено: 0 гостей с разным age в разных визитах) |
| `gender` | `first` | Атрибут гостя, не меняется |
| `discount` | `mean` | Средняя скидка по периодам — представительная оценка |
| `prev_stays` | `max` | Из CRM: число визитов до этого заезда. Одинаково во всех строках одного заезда, max безопасен |

**Результат:** 516 414 строк (схлопнуто 4 729 — это строки причины B).

## Этап 6 — Удаление служебных тарифов

**Решение:** удаляем строки, где `tariff` содержит `Служебный | СОТРУДНИКИ | Сотрудник`.

**Почему после схлопывания:** к этому моменту каждая строка уже представляет одного гостя в одном заезде. Фильтрация до схлопывания могла бы обрезать часть строк гостя, у которого один период — служебный, а другой — нет.

**Почему эти тарифы:** сотрудники санатория, проживающие как гости, не являются целевой аудиторией. Их паттерны (длительные проживания, нулевые или символические цены) исказят кластеры.

**Убрано:** 29 196 строк. Остаток: 487 218.

## Этап 7 — Фильтрация по nights (0 < nights ≤ 90)

**nights == 0:** заселение и выезд в один календарный день. Не полноценный заезд (дневной визит, посещение аквапарка, ошибка данных). 144 строки.

**nights > 90:** аномалии. Топ значений — 366, 377, 2024 ночей. Это либо годовые служебные записи (часть уже удалена), либо ошибки в датах (новогодний тур с датами 2019–2020 интерпретирован как 367 ночей). 19 строк, 0.09% данных.

**Порог 90:** выбран по анализу распределения. p99 = 24 ночи, p99.9 = 89 ночей. Максимальный реальный курс лечения в санатории — 21–28 дней. Всё свыше 90 — артефакты.

**Результат:** df_clean — **485 897 строк**.

## Этап 8 — Вычисление lead_time

**Определение:** `lead_time = check_in_dt - min(created_dt)` в днях, clip(lower=0).

**Бизнес-смысл:** горизонт бронирования. Разделяет два принципиально разных типа клиентов:
- **Короткий lead_time (0–14 дней):** спонтанные гости — реагируют на last-minute предложения. Их можно привлечь в периоды недозагруженности быстрыми акциями.
- **Длинный lead_time (> 3 месяцев):** планировщики — бронируют отпуск заранее. Для них работают акции раннего бронирования.

**min(created_dt) по (grp_norm, guest_id):** берём самую раннюю дату создания из всех тарифных строк — это момент, когда бронирование реально появилось в системе.

**clip(lower=0):** 3 057 строк (0.6%), где `created_dt > check_in_dt` — ретроспективный ввод. Менеджер оформил запись в систему уже после заезда гостя (телефонное или живое бронирование). Приравниваем к lead_time = 0 («забронировал в день заезда»).

**Данные:** медиана = 45 дней, p75 = 118 дней, max = 689 дней.

**Принцип группировки:** каждая категория = отдельный маркетинговый канал с отдельной стратегией заполнения недозагруженности:

| стандарт/оптима | стандарт, оптима, межсезон | Массовый сегмент |
| новогодний/банкет | новогодн, банкет, праздн | Праздничные пакеты |
| лечебный | лечени, оздоров, санатор | Санаторно-курортные путёвки |
| событие/группы | турнир, сборы, фестивал | Групповые мероприятия |
| социальный | школьн, студент, серебрян | Льготные социальные программы |
| акция/спецпредл | акция, аквапарк | Промо-предложения |

**Покрытие:** 90% строк попали в именованные категории, 10% — `other`.

## Этап 12 — Инжиниринг бинарных признаков и сезонности

### has_discount и discount_rate

Из колонки `discount` создаём оба производных признака:
- `has_discount = (discount > 0)` — бинарный флаг
- `discount_rate = discount.clip(0, 100).fillna(0)` — процент скидки

**В модель включаем только `discount_rate`.** Почему: `has_discount` — это просто бинаризация `discount_rate`. Корреляция между ними = 0.86. При евклидовой метрике KMeans оба признака давали бы двойной вес скидочному фактору, смещая кластеры. `discount_rate` содержит всю ту же информацию плюс величину скидки (5% и 50% — разные профили).

### has_children

`has_children = (children > 0)` — после распространения по этапу 4.

### is_weekend_in

`is_weekend_in = check_in_dt.dayofweek in [4, 5]` — заезд в пятницу или субботу.
**Бизнес-смысл:** отделяет weekend-туристов (2–3 ночи, спонтанный отдых) от weekday-гостей (длительные санаторные и корпоративные заезды). Будни — типичная зона недозагруженности, именно weekday-аудитория нужна для её заполнения.

### month_sin / month_cos

**Проблема числового месяца:** `check_in_month` как число нарушает цикличность. Декабрь (12) и январь (1) — соседние месяцы, но их числовая разность = 11, как между январём и декабрём. KMeans с евклидовой метрикой посчитает их «далёкими», что неверно.

**Решение:** циклическое кодирование через проекцию на единичную окружность:
```
month_sin = sin(2π · month / 12)
month_cos = cos(2π · month / 12)
```
Декабрь и январь оказываются рядом в двумерном пространстве (sin≈0, cos≈1). Для гостей без выраженной сезонности (приезжают летом и зимой) среднее sin/cos стремится к (0, 0) — это осмысленная точка «нет сезонного предпочтения».

### Исключённые признаки

| Признак | Причина исключения |
|---|---|
| `is_repeat` | Бинаризация `prev_stays`. Дублирует информацию |
| `is_partner` | Полностью перекрыт `tariff_group = льготный/корп` |
| `check_in_month` | Заменён на `month_sin/month_cos` |
| `has_discount` | Дублирует `discount_rate` (corr = 0.86) |

## Этап 14 — Агрегация до уровня гостя (df_for_model_claster)

**Почему нужна агрегация:** задача — сегментация клиентов, а не заездов. Гость, приехавший 5 раз, должен быть представлен одной строкой с усреднённым профилем — «кто этот человек в целом», а не «каким он был в конкретный визит».

**Правила агрегации по guest_id:**

| Признак | Агрегация | Обоснование |
|---|---|---|
| `stay_duration` | `mean` | Типичная длительность заезда для этого гостя |
| `lead_time` | `mean` | Типичный горизонт бронирования |
| `guest_age` | `first` | Статичный атрибут CRM — не меняется между визитами |
| `prev_stays` | `max` | Самое актуальное значение из CRM (накапливается со временем) |
| `discount_rate` | `mean` | Средняя ценовая чувствительность |
| `month_sin/cos` | `mean` | Круговое среднее — корректно для цикличных данных |
| `gender` | `first` | Неизменный атрибут гостя |

Причина: корреляция с `prev_stays` = 0.74. Обе метрики измеряют лояльность/частоту. `prev_stays` предпочтительнее — берётся из CRM и учитывает всю историю гостя, а не только наблюдаемый период.

## Этап 15 — Заполнение пропусков

**guest_age (28% пропусков):** заполняем медианой (40 лет).
**Почему медиана, не mean:** распределение возраста бимодальное (дети + взрослые). Медиана устойчива к этой форме, mean смещена.

**prev_stays (29% пропусков):** заполняем нулём.
**Почему 0, не медиана:** NaN в этом поле означает отсутствие записи в CRM — гость новый или его история не загружена. Ноль = «первичный гость, нет зафиксированных предыдущих визитов». Это содержательная, а не технически удобная замена.

**gender (19% пропусков):** заполняем категорией `'unknown'`.
**Почему не mode:** большинство гостей — женщины. Заполнить NaN «Женский» значит добавить ложные данные. `unknown` честно говорит: «не знаем».

## Этап 16 — Log1p трансформация перед StandardScaler

**Проблема:** `prev_stays` и `lead_time` имеют экстремальный правый хвост:

| Признак | p75 | p99 | max |
|---|---|---|---|
| prev_stays | 1 | 24 | 648 |
| lead_time | 118 дней | ~250 дней | 689 дней |

При `StandardScaler` (вычитает mean, делит на std) выброс с max=648 при p99=24 сильно смещает mean и std. Большинство гостей получают похожие масштабированные значения вблизи 0, а выброс уходит на сотни сигм — KMeans фактически игнорирует этот признак для большинства точек.

**Решение:** `log1p(x) = log(1 + x)` сжимает правый хвост, сохраняя:
- Монотонность (порядок значений не меняется)
- Ноль: `log1p(0) = 0` — гость без скидки остаётся нулём
- Интерпретируемость: различия между 1 и 10 визитами важнее, чем между 100 и 109

**Порядок применения:** пропуски → log1p → StandardScaler. Менять нельзя: log1p от NaN = NaN.
