## Бизнес-цель

Помочь санаторию управлять загрузкой номерного фонда:
1. **Сегментация клиентов** — разбить гостей на осмысленные кластеры
2. **Прогноз спроса** — предсказать загрузку по датам и типам номеров
3. **Связующая логика** — когда прогноз выявляет недозагруженность,
   система должна отвечать: какой сегмент клиентов привлечь и каким оффером

Практический выход: функция-рекомендатор
`get_target_segment(date, room_group, forecast_load) → сегмент + оффер`

### Ключевые факты о данных

- `is_valid_for_rfm == 1`: 521 143 строки — для кластеризации клиентов
- `is_valid_for_rfm == 0`: 44 673 строки — анонимы, использовать в прогнозе спроса
- Служебные тарифы (`Служебный|СОТРУДНИКИ|Сотрудник`) отфильтрованы
- Административные записи (`НЕ СЕЛИТЬ`) отфильтрованы

### Структура grp_norm (важно)

`grp_norm` = **ID бронирования**. Одно бронирование может содержать несколько строк
по двум разным причинам — их нельзя путать:

**Случай 1 — несколько гостей в одном бронировании (разные `guest_id`):**
Норма. Семья, пара, корпоративная группа едут вместе под одним `grp_norm`.
Пример: Фимин (1828) и Фимина (1829) — `grp_norm=294681`.
После `groupby(grp_norm, guest_id)` они остаются **двумя отдельными строками** — правильно,
каждый человек = отдельный клиент в кластеризации.

**Случай 2 — один гость с несколькими тарифными периодами (одинаковый `guest_id`):**
Технический артефакт. Один человек записан несколько раз внутри одного заезда
из-за смены тарифа или продления пребывания.
Пример: Фимин (1828) — 3 строки с разными датами внутри `grp_norm=294681`.
Именно эти строки схлопываются через `groupby(grp_norm, guest_id)`.

**Семантика `adults`, `children`, `composition`:**
Это атрибуты НОМЕРА, а не конкретного гостя. Записаны в ОДНОЙ строке из номера,
у остальных участников того же бронирования — `NaN`.

**Исправление в pipeline (до groupby):** `children` распространяется по всем гостям
одного `grp_norm` через `transform('max')` для бронирований ≤ 10 гостей (семьи).
Для групп > 10 человек не распространяем — слишком неоднородный состав.
Итог: has_children = 27% гостей (было 9% до исправления).

## Модель 1: Сегментация клиентов

### Статус: подготовка данных завершена ✅

**Основной ноутбук:** `my_cl.ipynb`

### Pipeline подготовки данных

1. Загрузка silver.csv → конвертация дат
2. Фильтр `is_valid_for_rfm == 1`
3. Вычисление `nights` по каждой строке ДО схлопывания
   (`normalize(check_out) - normalize(check_in)`, clip lower=0)
4. Вычисление `season` ДО схлопывания
5. Схлопывание по `(grp_norm, guest_id)` — один заезд = одна строка
6. Удаление служебных тарифов и `НЕ СЕЛИТЬ`
7. `df_clean`: фильтр `nights > 0 & nights <= 90` → **485 897 строк**
8. Инжиниринг признаков → `df_model` (485 897 × 14)
9. Агрегация до уровня гостя → `df_for_model_claster` (**436 492 гостя**)
10. Заполнение пропусков + log1p трансформация

### Финальный датасет: `df_for_model_claster`

**436 492 гостя × 14 колонок**

Матрица признаков `X_cols` (12 признаков):

| Признак | Тип | Агрегация | Бизнес-смысл |
|---|---|---|---|
| `stay_duration` | num | mean | Средняя длительность пребывания |
| `lead_time` | num | mean + **log1p** | Горизонт бронирования (дней до заезда) |
| `guest_age` | num | first | Возраст (median imputation) |
| `prev_stays` | num | max + **log1p** | Лояльность / число визитов из CRM |
| `discount_rate` | num | mean | Ценовая чувствительность (0–100%) |
| `month_sin` | num | mean | Циклическая сезонность (sin) |
| `month_cos` | num | mean | Циклическая сезонность (cos) |
| `tariff_group` | cat | mode | Ценовой сегмент / канал продаж (8 кат.) |
| `room_group` | cat | mode | Категория номера (5 кат.) |
| `gender` | cat | first | Пол (unknown imputation) |
| `has_children` | bin | max | Семейный сегмент (children распространён по grp_norm ≤10 гостей до схлопывания) |
| `is_weekend_in` | bin | mean | Доля заездов в выходные |

**Метаданные (не в X):** `guest_id`, `n_stays`

**Imputation (уже применён):**
- `guest_age` → median
- `prev_stays` → 0 (NaN = нет в CRM = первичный гость)
- `gender` → 'unknown'

**Log1p (уже применён):** `prev_stays` (max=648, p99=24), `lead_time` (max=689, p95=255)

### Почему признак включён / исключён

**Включены:**
- `month_sin/cos` вместо `check_in_month` — декабрь и январь соседние месяцы, числовой признак создаёт ложное расстояние 11 для KMeans
- `discount_rate` вместо `has_discount` — непрерывная мера ценовой чувствительности информативнее бинарной
- `prev_stays` вместо `is_repeat` — количественная мера лояльности vs бинарная

**Исключены:**
- `has_discount` — corr=0.86 с `discount_rate`, двойной вес в евклидовой метрике
- `is_repeat` — бинаризация `prev_stays`
- `is_partner` — перекрывается с `tariff_group` (льготный/корп)
- `n_stays` — corr=0.74 с `prev_stays`; `prev_stays` предпочтительнее (полная история CRM)
- `citizenship` — 99.9% РОССИЯ
- `currency` — одно значение

## Связующая логика (не начата)

После обучения обеих моделей строится **матрица аффинности**:

```python
affinity = df_guests.groupby(['cluster_id', 'room_group', 'check_in_month']).agg(
    n_guests      = ('guest_id', 'count'),
    avg_lead_time = ('lead_time', 'mean')
)
```

Использование: прогноз даёт недозагруженность в период T →
запрос к матрице: какой кластер исторически занимал эти номера в этот период
с `avg_lead_time`, позволяющим ещё успеть забронировать →
вывод: `cluster_id` + профиль + рекомендуемый тип оффера.
