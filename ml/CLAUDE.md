и прогнозирования спроса на туристические услуги (на примере санатория)»

---

## Бизнес-цель

Помочь санаторию управлять загрузкой номерного фонда:
1. **Сегментация клиентов** — разбить гостей на осмысленные кластеры
2. **Прогноз спроса** — предсказать загрузку по датам и типам номеров
3. **Связующая логика** — когда прогноз выявляет недозагруженность,
   система должна отвечать: какой сегмент клиентов привлечь и каким оффером

Практический выход: функция-рекомендатор
`get_target_segment(date, room_group, forecast_load) → сегмент + оффер`

### Ключевые факты о данных

- `is_valid == 1`: 521 143 строки — для кластеризации клиентов
- `is_valid == 0`: 44 673 строки — анонимы, использовать в прогнозе спроса
- Служебные тарифы (`Служебный|СОТРУДНИКИ|Сотрудник`) отфильтрованы
- Административные записи (`НЕ СЕЛИТЬ`) отфильтрованы

### Структура grp_norm (важно)

`grp_norm` = **ID бронирования**. Одно бронирование может содержать несколько строк
по двум разным причинам — их нельзя путать:

**Случай 1 — несколько гостей в одном бронировании (разные `guest_id`):**
Норма. Семья, пара, корпоративная группа едут вместе под одним `grp_norm`.
Пример: Фимин (1828) и Фимина (1829) — `grp_norm=294681`.
После `groupby(grp_norm, guest_id)` они остаются **двумя отдельными строками** — правильно,
каждый человек = отдельный клиент в кластеризации.

**Случай 2 — один гость с несколькими тарифными периодами (одинаковый `guest_id`):**
Технический артефакт. Один человек записан несколько раз внутри одного заезда
из-за смены тарифа или продления пребывания.
Пример: Фимин (1828) — 3 строки с разными датами внутри `grp_norm=294681`.
Именно эти строки схлопываются через `groupby(grp_norm, guest_id)`.

**Семантика `adults`, `children`, `composition`:**
Это атрибуты НОМЕРА, а не конкретного гостя. Записаны в ОДНОЙ строке из номера,
у остальных участников того же бронирования — `NaN`.

**Исправление в pipeline (до groupby):** `children` распространяется по всем гостям
одного `grp_norm` через `transform('max')` для бронирований ≤ 10 гостей (семьи).
Для групп > 10 человек не распространяем — слишком неоднородный состав.
Итог: has_children = 27% гостей (было 9% до исправления).

```
                 ↓
         Матрица аффинности
         cluster × room_group × месяц → avg_lead_time, n_guests
                 ↓
         Функция-рекомендатор:
         "Недозагруженность в период T → таргетировать кластер X"
```
