## Бизнес-цель

Помочь санаторию управлять спросом и загрузкой через прогноз активных бронирований:
1. **Сегментация клиентов** — разбить гостей на осмысленные кластеры
2. **Прогноз спроса** — предсказать число активных бронирований по датам
3. **Связующая логика** — когда прогноз выявляет недозагруженность,
   система должна отвечать: какой сегмент клиентов привлечь и каким оффером

Практический выход: функция-рекомендатор
`get_target_segment(date, room_group, forecast_load) → сегмент + оффер`

### Ключевые факты о данных

- `is_valid == 1`: 521 143 строки — для кластеризации клиентов и прогнозного ряда активных бронирований
- `is_valid == 0`: 44 673 строки — анонимные записи без `guest_id`, исключаются из ML-пайплайна; можно использовать отдельно только для sanity-check общего спроса
- Служебные тарифы (`Служебный|СОТРУДНИКИ|Сотрудник`) отфильтрованы
- Административные записи (`НЕ СЕЛИТЬ`) отфильтрованы

### Структура grp_norm (важно)

`grp_norm` = **ID бронирования**. Одно бронирование может содержать несколько строк
по двум разным причинам — их нельзя путать:

**Случай 1 — несколько гостей в одном бронировании (разные `guest_id`):**
Норма. Семья, пара, корпоративная группа едут вместе под одним `grp_norm`.
Пример: Фимин (1828) и Фимина (1829) — `grp_norm=294681`.
После `groupby(grp_norm, guest_id)` они остаются **двумя отдельными строками** — правильно,
каждый человек = отдельный клиент в кластеризации.

**Случай 2 — один гость с несколькими тарифными периодами (одинаковый `guest_id`):**
Технический артефакт. Один человек записан несколько раз внутри одного заезда
из-за смены тарифа или продления пребывания.
Пример: Фимин (1828) — 3 строки с разными датами внутри `grp_norm=294681`.
Именно эти строки схлопываются через `groupby(grp_norm, guest_id)`.

### Пайплайн данных (`pipeline_clustering.ipynb`)

`silver.csv` → `pipeline_clustering.ipynb` → `artifacts/df_for_model_cluster.parquet` (248 547 гостей)

**Шаги:**
1. Фильтр `is_valid=1` (оставляем 521 143 строки)
2. Исключение ковидного периода 2020–2021 (`USE_FULL_HISTORY=False`, данные с 2022-01-01)
3. Признаки ДО схлопывания: `nights` (check_out - check_in), `season`, распространение `children` по `grp_norm` для малых групп (≤10 чел.) через `transform('max')`
4. Схлопывание по `(grp_norm, guest_id)` — устраняет технический дубль тарифных строк
5. Удаление служебных тарифов (`Служебный|СОТРУДНИКИ|Сотрудник|НЕ СЕЛИТЬ`), ФСС/корпоративных путёвок, `nights==0` и `nights>90`
6. Инжиниринг признаков: `lead_time`, `guest_age`, `tariff_group` (628→8 групп), `room_group` (24→5), `has_children`, `pct_weekend_in`, `is_treatment`, `pct_summer/autumn/winter/spring`, `price_tier` (ЛАЙТ=1, ОПТИМА=2, УЛЬТРА=3)
7. Агрегация до уровня гостя (1 гость = 1 строка): mean/max по заездам
8. Импутация: `guest_age` — медиана по `tariff_group`; `prev_stays` — 0 (первичный гость)
9. log1p-трансформация: `lead_time`, `prev_stays` (длинные хвосты → без log1p KMeans игнорирует другие признаки)
10. Препроцессор (`ColumnTransformer`): already_log1p → StandardScaler; needs_log1p → log1p+StandardScaler; числовые и бинарные → StandardScaler

**Сохраняет в `artifacts/`:** `df_for_model_cluster.parquet`, `preprocessor.joblib`, `feature_meta.joblib`, `X_scaled.npy`

**Алгоритм:** K-Means (k-means++, n_init=20, random_state=42)  
**k = 6**, **8 признаков** (чисто поведенческих + мотивация + бюджет, без сезонных pct_*)

**Признаки модели:**

| Признак | Трансформация | Обоснование |
|---|---|---|
| `lead_time` | уже log1p → StandardScaler | Горизонт бронирования |
| `prev_stays` | уже log1p → StandardScaler | Лояльность из CRM |
| `stay_duration` | log1p → StandardScaler | Длительность пребывания |
| `guest_age` | StandardScaler | Возраст |
| `pct_weekend_in` | StandardScaler | Паттерн дня заезда (будни vs выходные) |
| `price_tier` | StandardScaler | Бюджет: ЛАЙТ=1, ОПТИМА=2, УЛЬТРА=3 |
| `has_children` | StandardScaler | Семейный гость |
| `is_treatment` | StandardScaler | Мотивация: оздоровление vs отдых |

**Не входят в модель** (хранятся в df для профилирования):  
`pct_summer`, `pct_autumn`, `pct_winter`, `pct_spring` → идут в матрицу аффинности

## Модель 2: Прогноз активных бронирований

**Статус:** завершена. Финальная версия — baseline Prophet из `forecasting/02_prophet_baseline.ipynb`, потому что улучшенные варианты не дали выигрыша по MAPE.

### Порядок запуска

1. `forecasting/01_timeseries_build.ipynb` — строит ежедневный ряд `df_daily.parquet`.
2. `forecasting/02_prophet_baseline.ipynb` — обучает baseline Prophet и сохраняет `forecast_baseline.parquet`.
3. `forecasting/03_prophet_improved.ipynb` — сравнивает альтернативы Prophet и сохраняет `forecast_best.parquet`.
4. `forecasting/04_underload_detection.ipynb` — формирует даты риска.
5. `forecasting/05_backtesting.py` — rolling backtesting baseline-модели по нескольким cutoff.

### Целевая переменная и учет емкости

`y` = количество активных бронирований в день.  
Расчет строится через разворачивание интервала `check_in_dt` → `check_out_dt` и агрегацию `nunique(grp_norm)` по дате. `grp_norm` трактуется как ID бронирования, поэтому такая агрегация убирает технические дубли строк внутри одного бронирования.

Так как в 2024 году изменилась верхняя граница ряда активных бронирований, абсолютные значения `y` нормализуются на оценочную емкость:

`capacity_est(year) = 99-й перцентиль дневного y внутри года`

`load_rate = y / capacity_est`

Prophet обучается на `load_rate`, после чего прогноз переводится обратно в активные бронирования:

`forecast_load = yhat_rate * capacity_est`

Оценочная емкость по годам:

| Год | max y | p99 y / capacity_est | mean y |
|---:|---:|---:|---:|
| 2022 | 980 | 961 | 376.4 |
| 2023 | 992 | 982 | 522.0 |
| 2024 | 861 | 824 | 419.1 |
| 2025 | 813 | 779 | 435.8 |

### Настройки финального Prophet

Финальная модель использует:

| Компонент | Настройка |
|---|---|
| Тренд | стандартный piecewise linear trend Prophet |
| Годовая сезонность | `yearly_seasonality=True` |
| Недельная сезонность | `weekly_seasonality=True` |
| Дневная сезонность | `daily_seasonality=False` |
| Режим сезонности | `seasonality_mode='additive'` |
| Календарь событий | `holidays=build_holidays_df()` |
| Горизонт прогноза | 365 дней |
| Интервал неопределенности | `interval_width=0.90` |

Календарь событий включает официальные праздники РФ и школьные каникулы из `forecasting/holidays_ru.py`. Для части праздников заданы `lower_window` / `upper_window`, поэтому Prophet учитывает не только сам день события, но и соседние дни.

### Что проверялось в improved-версии

| Версия | Изменение | MAPE |
|---|---|---:|
| baseline | holidays + yearly/weekly seasonality | 19.78 |
| v1_multiplicative | multiplicative seasonality | 20.24 |

Вывод: baseline остался финальным, потому что все дополнительные варианты ухудшили MAPE на holdout.

### Lag-признаки

Явные авторегрессионные lag-признаки (`y_lag_7`, `rolling_mean_7` и т.п.) в текущей версии не используются. Это сделано осознанно: для горизонта 365 дней будущие lag-и пришлось бы получать рекурсивно, что усложняет пайплайн и повышает риск leakage при оценке. Текущая модель опирается на тренд, календарные сезонности и календарь событий.

### Альтернативная модель: XGBoost

Дополнительно протестирована модель `XGBRegressor` из `forecasting/06_xgboost_backtesting.py`.

**Признаки:**
- трендовый индекс `time_idx`;
- календарь: месяц, квартал, неделя, день недели, день года, начало/конец месяца;
- циклическая сезонность: `sin/cos` для года и недели;
- события: окно официальных праздников, школьные каникулы, длинные выходные;
- лаги: `lag_7`, `lag_14`, `lag_28`.

**Параметры:**

```python
{
    "n_estimators": 300,
    "learning_rate": 0.03,
    "max_depth": 3,
    "min_child_weight": 3,
    "subsample": 0.85,
    "colsample_bytree": 0.85,
    "reg_lambda": 3.0,
    "reg_alpha": 0.1,
    "objective": "reg:squarederror",
    "eval_metric": "rmse",
    "random_state": 42,
}
```

Для исключения утечки данных многодневный прогноз строится рекурсивно: прогноз следующего дня добавляется в историю и используется при расчете лагов для следующих дат.

Rolling backtesting XGBoost:

| Горизонт | MAE | RMSE | MAPE |
|---:|---:|---:|---:|
| 30 дней | 68.14 | 83.65 | 21.98 |
| 60 дней | 66.08 | 80.30 | 19.55 |
| 90 дней | 67.34 | 84.74 | 20.08 |

Вывод: XGBoost снижает MAE/RMSE на части горизонтов, но по MAPE не превосходит capacity-adjusted Prophet. Поэтому финальной моделью остается Prophet, а XGBoost сохраняется как альтернативный эксперимент.

### Недозагрузка / риск

Риск считается относительно сезонной цели по нормализованному ряду:

`target_rate_season = 60-й перцентиль исторического load_rate внутри сезона`

`target_load_season = target_rate_season * capacity_est`

Дата считается рискованной, если:

`forecast_load < target_load_season`

## Модель 3: Рекомендательная логика

**Статус:** завершена как rule-based scoring, а не обученная рекомендательная модель.

Для каждой даты риска строится скор по 6 сегментам:

`score = 0.50 * season_score + 0.20 * value_score + 0.15 * stability_score + 0.15 * lead_score`

Компоненты:

| Компонент | Смысл |
|---|---|
| `season_score` | насколько сегмент исторически активен в текущий сезон |
| `value_score` | нормализованный `avg_price_tier` |
| `stability_score` | обратная доля скидок |
| `lead_score` | горизонт планирования сегмента |

Финальная витрина рекомендаций сохраняется в `recommender/artifacts_recommender/recommendations.parquet` и `recommendations.csv`.
